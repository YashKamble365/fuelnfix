Core Architecture

FuelNFix is a real-time, location-based service built on the MERN stack. Unlike traditional CRUD apps, it relies heavily on:

  • Geospatial Queries: MongoDB $near and 2dsphere indexes to find nearest providers.
  • Event-Driven State: Socket.IO events drive the UI state (e.g., "Request Accepted" instantly updates the user's screen without a refresh).
  • State Machine Wrappers: The frontend wizards use complex internal state machines to guide users through multi-step processes.

Backend Architecture

Server Entry Point

server/index.js

This file is the "Brain". It bridges HTTP requests (Express) and WebSocket events (Socket.IO). Pay attention to how it handles the provider_location_update event.

const express = require('express');
const { Server } = require("socket.io");
const mongoose = require('mongoose');

// 1. Database Connection
mongoose.connect(process.env.MONGO_URI).then(() => {
    // Auto-seed default service rates if DB is empty
    ServiceRate.seedDefaults(); 
});

// 2. Real-Time Logic (The "Uber" Tracking)
io.on('connection', (socket) => {
    
    // A. Provider Report Location
    socket.on('provider_location_update', async ({ roomId, userId, location }) => {
        
        // OPTIMIZATION: Broadcast to room immediately for smooth UI animation
        if (roomId) io.to(roomId).emit('track_provider', location);

        // PERSISTENCE: Save to DB for "Search Nearby" functionality
        // We update this less frequently in production, but real-time here for demo
        if (userId) {
            await User.findByIdAndUpdate(userId, {
                liveLocation: { type: 'Point', coordinates: [location.lng, location.lat] },
                isOnline: true
            });
        }
    });

    // B. Join Request Room (Privacy Isolation)
    socket.on('join_room', (roomId) => {
        socket.join(roomId); // Only User and Provider in this request allow to talk
    });
});

Data Models

The User Model (Provider & Customer)

server/models/User.js

We use a single User model for both customers and providers to allow role switching and simpler auth. The distinction is made via the role field.

const UserSchema = new mongoose.Schema({
    role: { type: String, default: 'user' }, // 'user', 'provider', 'admin'
    
    // Geolocation Indexing (CRITICAL)
    // '2dsphere' index allows efficient Earth-surface queries
    location: {
        type: { type: String, enum: ['Point'], default: 'Point' },
        coordinates: { type: [Number] } // [Longitude, Latitude] - Mongo requires Lng first!
    },
    
    // Provider Specifics
    providerCategory: [{ type: String, enum: ['Mechanic', 'Fuel Delivery'] }],
    isVerified: { type: Boolean, default: false },
    verificationDocs: { shopPhoto: String, businessProof: String }
});

UserSchema.index({ location: '2dsphere' }); // Must enable this in MongoDB Atlas

The Service Pricing Model

server/models/ServiceRate.js

Allows Dynamic Pricing configurable by the Admin. Instead of hardcoding prices in code, we fetch them from this collection.

const ServiceRateSchema = new mongoose.Schema({
    serviceName: { type: String, unique: true }, // "Flat Tire"
    basePrice: { type: Number }, // e.g., 300
    pricePerKm: { type: Number }, // e.g., 20/km
    pricePerLitre: { type: Number }, // Only for Fuel Delivery
    category: { type: String, enum: ['Mechanic', 'EV Support', 'Fuel Delivery'] }
});

The Brain: Matching Algorithm

Request Controller

server/controllers/requestController.js

This is where the magic happens. We don't just return all providers; we filter by service capabilities, check verification status, and calculate dynamic pricing based on distance.

exports.searchProviders = async (req, res) => {
    const { serviceName, originLat, originLng } = req.body;

    // 1. The Proximity Search
    const providers = await User.find({
        role: 'provider',
        isVerified: true,
        isOnline: true,
        location: {
            $near: {
                $geometry: { type: "Point", coordinates: [originLng, originLat] },
                $maxDistance: 50000 // 50km Hard Limit
            }
        },
        services: { $elemMatch: { name: serviceName, active: true } }
    });

    // 2. The Pricing Engine
    const results = providers.map(p => {
        const distance = calculateHaversine(originLat, originLng, p.location);
        const rate = await ServiceRate.findOne({ name: serviceName });
        
        return {
            provider: p,
            distance: distance,
            // Formula: Base + (Dist * Rate)
            totalEstimate: rate.basePrice + (distance * rate.pricePerKm)
        };
    });

    res.json(results);
};

The Admin Dashboard

AdminDashboard.jsx

client/src/pages/AdminDashboard.jsx

A massive component (1500+ lines) acting as the Control Center. It aggregates data from multiple endpoints.

1. Data Fetching Strategy

We use Promise.all to fetch 7 different datasets concurrently on load. This prevents a "waterfall" of requests and ensures the dashboard pops in with data all at once.

const fetchAllData = async () => {
    const [pending, verified, users, analytics, pricing, broadcast] = await Promise.all([
        api.get('/pending-providers'),
        api.get('/verified-providers'),
        api.get('/users'),
        api.get('/analytics'),
        api.get('/pricing'),
        api.get('/announcements')
    ]);
    // Batch state updates to minimize re-renders
    setStats({ ... });
    setPendingProviders(pending.data);
};

2. Provider Verification Flow

The dashboard renders cards for each pending provider. Clicking "Verify" sends a PUT request to /api/admin/verify, which toggles the `isVerified` flag on the User model, instantly allowing them to appear in search results.

3. Broadcast System

The Admin can send "Announcements". These are saved to DB and *immediately emitted via Socket.IO* to all connected clients. This is how we send "Heavy Rain Alerts" etc.
